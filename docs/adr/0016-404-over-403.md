# ADR-0016: Return 404 (not 403) for records the caller can't see

**Status:** Accepted  ·  **Date:** 2026-07-24

## Context

A 403 for an unrelated resource confirms the resource exists — the attacker learns something they shouldn't. Enumerating IDs against a 403-returning endpoint reveals the ID space.

## Decision

- **401** — missing or invalid access token
- **403** — authenticated with the wrong role, on an endpoint you know you can't hit (`POST /jobs` as a student, etc.). Nothing sensitive is leaked; the endpoint's existence is public.
- **404** — you have permission to hit the endpoint but not to see this particular row. Do not confirm the record exists.

## Consequences

- Ownership scoping is applied **inside the query**, not after fetching → a scoped `findFirst` returns `null` for both "doesn't exist" and "not yours"
- Enumerating IDs is uninformative
- Verified by `tests/auth.test.js` — student A requesting student B's application gets `404` with `RESOURCE_NOT_FOUND`
